Git and Github

How did anybody ever do anything before Github?
https://twitter.com/pikesley/status/303100904037556225
Does everybody understand what we mean by Version Control? For those of you who don't, Version Control is a bit like Google Documents, in that it retains past versions of our code, and also allows us to collaborate very easily. Git is our current Weapon Of Choice - in previous tools like Subversion, operations such as branching and merging were extremely expensive and painful. With Git, these things become a positive pleasure.

Git

http://git-scm.com

  • Distributed version control
  • Extremely cheap branching
  • Extremely good merging
Github has changed pretty much everything. It's difficult to conceive now of how we did ANYTHING before Github existed. As an example, on Good Friday this year I was sitting at my flat, collaborating over Github with some guy in Brazil to fix up somebody else's tool. The future is here, and it's amazing.

GitHub

http://github.com

  • Git hosting
  • Issue tracking
  • Wikis for documentation

Pull Requests

Fork (or branch) and Pull

How GitHub Uses GitHub to Build GitHub

I've added something to your code, and I want you to merge it upstream.

Workflow

  • Clone the repository
  • Create a branch
  • Write some code (with tests!)
  • Publish branch
  • Open a pull request back to master
GitHub are calling this "GitHub Flow". Important aspects: short-lived branches, minimal merges. We shall return to this theme later
Here's an example, here James has opened a PR on some of Jeni's code
https://github.com/theodi/member-directory/pull/72
Here we can see the files that have been changed
https://github.com/theodi/member-directory/pull/72
And here are the actual changes within the code, all highlighted
https://github.com/theodi/member-directory/pull/72
A PR is not just about "here's my code", it's also way to facilitate a discussion. We usually open a PR on the first commit, and as you can see here, a further commit has been made halfway through, with a description of what's changed. It's like a comment thread, but without the shouting.
https://github.com/theodi/member-directory/pull/72
And then the PR is merged. And this is the last place any humans touch the code, but more of that later...

Sam Continuous deployment has been a Holy Grail for me for a long time, but it's always been too scary to attempt. We're very fortunate at the ODI in that we've started from scratch, so this has been baked in from the beginning.

Continuous Deployment

Jenkins is our Continuous Integration robot, it checks out the code and runs the tests just the same as we would do on our laptops
http://jenkins.theodi.org
And we can now hook it into Github so we can see in the PR that this branch has passing tests
Previously our Jenkins setup was a bit Heath-Robinson, but we recently discovered a lovely tool for managing this stuff. So we now have a robot to manage what Jenkins does, and then publish documentation and so on. This is a work in progress - it's all a work in progress - and it will do more stuff in the future. And we're always flexible - we've recently been trialling Travis, which is CI-as-a-Service.

Master Builder

https://github.com/theodi/master-builder

  • Written using the Jenkins Job DSL
  • Monitors all branches of all our projects
  • Creates all the Jenkins jobs
  • Tags selected branches
  • Also does:
    • Coverage
    • Documentation
    • Github status
    • Notifications
So Jenkins has told us that the code is good to go, so we need to get it out into the wild. In the bad old days, we would own a bunch of actual servers, which we'd have to look after and care for over a long period of time, and deploy to them probably by hand, which is error-prone and unpleasant. This is the classic Snowflake anti-pattern. Not any more. Now we have ephemeral cloud servers, and deployment is in the hands of the Chef robots.

Treat your servers as cattle, not as pets

No more snowflakes!

We can now describe our infrastructure as code. Executable Infrastructure is a beautiful thing. This ties in nicely with the short-lived, frequently-merged branches from the Git Flow model, and it means we can deploy to production many, many times per day without ever thinking about it. As soon as the PR is merged, the robots take over. Deployments should not be like going to the dentist - we've done this on a Friday afternoon without any real terror.

Chef

http://www.opscode.com/chef/

  • Infrastructure == code
  • Continuous deployment
  • Deploys latest release
  • Holds the keys!
  • Code live in 10 minutes
  • No human intervention
  • cucumber-chef
And it's all testable. Test-driven Executable Infrastructure is an even more beautiful thing. This describes what I want this server to do, which provides a lovely safety-harness when I'm making changes. If your machines are Identical, Repeatable and Disposable, you'll have a good time.
